Las tres fases de la normalización

Fecha: 6 de Febrero, 2026

1. Objetivo y Contexto

Objetivo: Aplicar el proceso teórico de normalización a un conjunto de datos desestructurado ("sábana de datos") para convertirlo en un esquema relacional optimizado. El fin es eliminar redundancias y prevenir anomalías de inserción, actualización y borrado.

Contexto: En el mundo real, los datos suelen llegar en formatos planos (Excel, CSV) no normalizados. Diseñar una base de datos eficiente requiere descomponer estos datos en tablas lógicas conectadas por llaves foráneas antes de escribir una sola línea de código SQL.

2. Conceptos y Profundización

La normalización se basa en reglas formales llamadas "Formas Normales" (FN):

  • Primera Forma Normal (1FN): Exige atomicidad. Cada celda debe contener un único valor y no debe haber grupos repetidos de columnas. Se identifica una llave primaria.
  • Segunda Forma Normal (2FN): Se basa en la dependencia funcional total. Elimina las dependencias parciales; es decir, todos los atributos que no son llave deben depender de la totalidad de la llave primaria (crucial en llaves compuestas).
  • Tercera Forma Normal (3FN): Elimina las dependencias transitivas. Ningún atributo no-llave debe depender de otro atributo no-llave. "La llave, toda la llave y nada más que la llave".
  • Anomalías: Sin normalización, borrar un pedido podría borrar accidentalmente la información del cliente (anomalía de borrado), o actualizar la dirección de un cliente requeriría cambios en múltiples filas (anomalía de actualización).

Esquema SQL Resultante (DDL):

CREATE TABLE clientes(
    id_cliente int primary key,
    nom_cliente varchar(100),
    id_estado int not null,
    FOREIGN KEY(id_estado) REFERENCES estados(id_estado)
);

CREATE TABLE articulos_ordenes(
    id_orden int not null,
    num_item int not null,
    PRIMARY KEY(id_orden, num_item), -- Llave compuesta
    FOREIGN KEY(id_orden) REFERENCES ordenes(id_orden),
    FOREIGN KEY(num_item) REFERENCES articulos(num_item)
);

5. Conclusiones

Aprendizaje: La normalización no es solo teoría académica; es vital para la integridad de los datos. Aprendí que una llave primaria compuesta (`id_orden`, `num_item`) es necesaria para resolver relaciones de muchos a muchos en tablas de detalle.

Problemas y Soluciones: Identificar las dependencias transitivas fue complejo (ej. Estado dependiendo de Cliente). La solución fue crear tablas "catálogo" separadas para evitar repetir textos como "California" o "Wisconsin" múltiples veces.

Mejoras Futuras: Analizar casos donde la des-normalización sea necesaria por rendimiento (ej. Data Warehousing), entendiendo el costo-beneficio.
