Uso de PostgreSQL en la VM

Fecha: 27 de Enero, 2026

1. Objetivo y Contexto

Objetivo: Instalar y configurar PostgreSQL como segundo Sistema Gestor de Bases de Datos en la misma infraestructura. El propósito es contrastar las diferencias sintácticas y de arquitectura frente a MariaDB/MySQL, enfocándose en la gestión de tipos de datos estrictos y la interfaz de línea de comandos `psql`.

Contexto: En la industria, es común encontrar arquitecturas políglotas. Mientras MariaDB suele ser la opción para stacks web rápidos (LAMP), PostgreSQL se prefiere en entornos que requieren integridad referencial estricta, soporte para datos geoespaciales o tipos de datos complejos (JSONB, Arrays).

2. Conceptos y Profundización

  • ORDBMS (Object-Relational DBMS): PostgreSQL no es solo relacional; soporta características orientadas a objetos como herencia de tablas y tipos de datos personalizados. Es conocido por su cumplimiento estricto de los estándares ANSI SQL.
  • Arquitectura de Procesos: A diferencia de MySQL que usa hilos (threads) por conexión, Postgres crea un proceso nuevo por cada conexión, lo que le da estabilidad robusta pero requiere una gestión de memoria diferente.
  • Comandos Meta (`\`): La herramienta CLI `psql` utiliza comandos precedidos por backslash que no son SQL estándar, como `\l` (listar bases), `\c` (conectar), `\dt` (listar tablas). Esto es una diferencia clave de usabilidad frente al monitor de MySQL.
  • Tipado Fuerte: Postgres es menos permisivo que MariaDB. Por ejemplo, el manejo de comillas dobles para identificadores y comillas simples para literales de cadena es estricto, así como la sensibilidad a mayúsculas/minúsculas.

4. Entregables

Instrucciones rápidas:

$ sudo dnf install postgresql-server -y
$ sudo postgresql-setup --initdb
$ sudo systemctl start postgresql
$ sudo su - postgres
$ psql
postgres=# CREATE DATABASE biblioteca;
postgres=# \c biblioteca
biblioteca=# CREATE TABLE Alumnos(
    matricula integer generated by default as identity primary key,
    nombre varchar(100) not null
);

5. Conclusiones

Aprendizaje: La adaptación al cambio de sintaxis fue el mayor reto. Aprendí que `AUTO_INCREMENT` es un "dialecto" de MySQL, mientras que `GENERATED BY DEFAULT AS IDENTITY` es el estándar SQL que usa Postgres. También valoré la potencia de la herramienta `psql` para administración rápida.

Problemas y Soluciones: Postgres normaliza automáticamente los nombres de tablas y columnas a minúsculas a menos que se usen comillas dobles. Esto causó errores de "relation not found". La solución fue adherirse a la convención de usar siempre snake_case en minúsculas para los identificadores.

Mejoras Futuras: Explorar tipos de datos avanzados exclusivos de Postgres, como los campos JSONB para almacenar datos semi-estructurados.
